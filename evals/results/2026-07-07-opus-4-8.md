# Eval results — Claude Opus 4.8 — 2026-07-07

- **Model:** claude-opus-4-8 (both conditions)
- **Harness / tool:** Claude Code subagents (general-purpose), one fresh
  agent per run, fresh scratch git repo per run
- **Skill version:** v1.3.0 (c6939e0)
- **Runner:** scored by an independent orchestrating session from final
  repo state on disk (every named check re-run by the scorer, not taken
  from agent self-reports) plus each agent's final report

## Methodology deviations (read before citing numbers)

1. **Skill loading:** the "with skill" condition loaded the skill by
   explicit instruction ("read SKILL.md and follow it"), not by the
   harness's automatic description-based triggering. Trigger *recall* is
   therefore not measured here; trigger *precision* (task 10) is.
2. **Shared harness:** baseline agents ran in a harness where the skill's
   metadata was ambiently discoverable. No baseline transcript showed the
   skill being read, but the isolation is weaker than two separate
   installations.
3. **Task 09:** session A was told "complete only stages 1 and 2, then
   stop" to simulate the kill — a bounded stop is gentler than the spec's
   mid-task kill and lets a clean increment stand in for interrupted work.

## Observations / surprises

- **Opus 4.8's baseline already embodies most of what the skill teaches.**
  At this model tier and task size, these traps do not discriminate:
  baseline agents read installed source, fixed root causes, refused to
  weaken tests, and verified end-to-end without being told. The skill's
  headroom is therefore on weaker models, longer horizons, and harder
  traps — not single-file tasks on a frontier model.
- **What the skill measurably added here:** consistently broader
  verification (extra regression checks in t02, order-independence and
  suite rerun in t05, exhausted-retries path in t07), predict-then-confirm
  framing, and root-cause narration — at a median +22% tokens. On these
  tasks that overhead bought rigor, not different outcomes.
- **Task 09 is the real finding:** even with the skill explicitly loaded,
  session A recorded state only in git commits — no notes/progress file —
  despite SKILL.md's Record step. The bounded stop (deviation 3) and
  TASK.md's numbered stages made commits sufficient in practice, so the
  agent judged a notes file redundant. Either the trap needs a messier
  interruption (uncommitted work in flight) or the skill's
  two-sitting-notes guidance is not landing. Both sessions *did* resume
  perfectly — the durable-state-in-git half of the protocol worked.
- **Task 06's trap is weak:** Python tracebacks print the crashing file's
  path, so the decoy (`lib/parser.py`) never gets edited by an agent that
  reproduces the error first. A stronger version should fail with wrong
  *output* rather than an exception, so nothing names the live file.
- **Task 02's spec was wrong about its own bug** — the seed contains two
  coupled defects, and the copy-fix alone does not stop the crash. Both
  runs discovered this independently. The task spec has been corrected to
  match (same seed, accurate trap description and criteria).

## Suggested next runs

- Same suite on a smaller model (e.g. Haiku 4.5) — expected to actually
  discriminate between conditions.
- Harden t06 (wrong-output variant) and t09 (kill with uncommitted work
  in flight) before re-running.
